ТЕХНИЧЕСКОЕ ЗАДАНИЕ (ТЗ) на создание макета среды разработки для систем распределённого управления по IEC 61499 (аналог Eclipse 4diac IDE)
Разработчик: Дубровин Н.Д. (гр. М3О-321Б-23), Чикунов Ю.В. (гр. М3О-321Б-23)

1. Введение
1.1. Техническое задание определяет цели, требования и порядок разработки макета среды разработки (МСР) для создания, редактирования и развёртывания приложений распределённого управления в соответствии со стандартом IEC 61499. МСР является аналогом Eclipse 4diac IDE и предназначена для моделирования графов функциональных блоков (ФБ), аппаратных конфигураций и библиотек типов.
1.2. Объём работ включает разработку прототипа на базе Eclipse RCP (Rich Client Platform) с поддержкой XML-формата проектов (.sys), графического редактора и протокола TCP для взаимодействия со средой исполнения (аналог FORTE). Разработка ведётся на языке Java с использованием библиотек Eclipse Modeling Framework (EMF) и Graphiti для визуализации.

2. Основания для разработки 
2.1. Разработка обусловлена необходимостью создания открытой среды для обучения и прототипирования систем промышленной автоматизации по IEC 61499, аналогичной Eclipse 4diac. Текущие коммерческие инструменты (например, ISaGRAF) не обеспечивают открытости и интеграции с OPC UA/MQTT.
2.2. Нормативные основания: ГОСТ Р ИСО/МЭК 61499-1-2011 (распределённые системы управления); Eclipse Public License 2.0 для базового кода 4diac.
2.3. Исходные данные: Анализ Eclipse 4diac (документация на https://eclipse.dev/4diac/documentation/); лабораторная работа "Работа с требованиями" (Максимов А.Н., 2024).

3. Назначение 
3.1. МСР предназначена для:
•	Создания и редактирования проектов IEC 61499 (приложения, аппаратные конфигурации, библиотеки ФБ).
•	Визуализации графов связей ФБ на экране.
•	Генерации XML-файлов проектов (.sys) и загрузочных файлов (.fboot).
•	Развёртывания проектов в среду исполнения по TCP-протоколу, совместимому с 4diac.
•	Мониторинга и отладки переменных ФБ в реальном времени.
3.2. МСР будет использоваться в образовательных целях (лабораторные работы) и для прототипирования промышленных систем (Industrie 4.0). Целевая аудитория: студенты, инженеры автоматизации.
3.3. Ожидаемые результаты: Рабочий прототип IDE с тестовым проектом "Hello World", интегрированный с OPC UA клиентом для чтения/записи тегов.

4. Требования к программе 
4.1. Требования к функциональным характеристикам 
    4.1.1. Загрузка и сохранение проектов в формате XML (.sys), содержащем приложения, конфигурации и библиотеки ФБ (аналогично 4diac). 
    4.1.2. Графический редактор для отображения и редактирования связей ФБ (drag-and-drop, zoom, undo/redo). Поддержка базовых ФБ: STRING2STRING, APPEND_STRING, EMB_RES, ANY_CONSOLE. 
    4.1.3. Назначение ФБ на ресурсы (EMB_RES) и генерация .fboot для загрузки в runtime. 
    4.1.4. Развёртывание по TCP (порт 11740): Отправка команд CREATE, CONNECTION, START. 
	4.1.5. Мониторинг: Чтение/запись переменных по TCP; визуализация в таблице (watch). 
	4.1.6. Интеграция OPC UA клиента: Подключение к серверу, просмотр тегов, периодическое чтение/запись (библиотека freeopcua или open62541).
	4.1.7. Тестирование: Автоматический запуск тестового приложения с выводом "Hello world". Функциональность должна обеспечивать производительность: Загрузка проекта < 5 сек, развёртывание < 10 сек (на ПК Intel i5+).
4.2. Требования к составу и параметрам технических средств 
	4.2.1. Среда разработки: Windows 10/11 (64-bit), Java JDK 17+, Eclipse RCP 2023-12. 
	4.2.2. Минимальные характеристики: CPU 2 ГГц, RAM 8 ГБ, HDD 500 МБ. 
	4.2.3. Состав: Основной модуль (IDE), плагины (Graphiti для графики, EMF для XML), внешние библиотеки (open62541 для OPC UA). 
	4.2.4. Установка: Портативная (zip-архив), без реестра. Автозапуск тестов при первом запуске. 
	4.2.5. Надёжность: Автосохранение каждые 5 мин; обработка ошибок TCP (retry 3 раза).
4.3. Требования к информационной и программной совместимости 
	4.3.1. Совместимость с IEC 61499: Поддержка XML-schema 4diac (.sys, .fboot). 
	4.3.2. Протоколы: TCP для развёртывания/мониторинга (совместим с FORTE); OPC UA 1.04 (UA-TCP). 
	4.3.3. Интеграция: Экспорт в FORTE (C++ runtime); импорт библиотек из Eclipse 4diac. 
	4.3.4. Портативность: Работает с runtime на Linux/Windows (cross-compile). Нет зависимостей от проприетарного ПО. 
	4.3.5. Безопасность: Шифрование TCP (опционально TLS); доступ только к localhost по умолчанию.

5. Требования к программной документации
5.1. Программная документация должна быть разработана в соответствии с ГОСТ 19.101-77, ГОСТ 19.201-78 и ГОСТ 19.202-78.
5.2. Состав документации:
•	Техническое задание (настоящий документ).
•	Программа и методика испытаний (ПМИ) — описание сценариев тестирования (запуск "Hello World", мониторинг переменных, OPC UA клиент).
•	Руководство пользователя — пошаговые инструкции по установке, созданию проекта, развёртыванию, мониторингу (с скриншотами).
•	Руководство программиста — описание архитектуры, API для плагинов, форматов .sys и .fboot.
•	Описание применения — примеры использования в учебных задачах.
5.3. Формат и оформление:
•	Все документы в формате .docx и .pdf (экспорт из Word).
•	Шрифт: Times New Roman, 14 кегль, интервал 1.5.
•	Нумерация разделов и рисунков по ГОСТ 19.003-80.
•	Обложка по ГОСТ 19.104-78 (название, разработчик, дата, гриф).
5.4. Поставка:
•	Документация включается в дистрибутив МСР (папка docs/).
•	Версия документации синхронизирована с версией ПО (v1.0).
•	Электронная подпись (опционально) для ПМИ и ТЗ.
